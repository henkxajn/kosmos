## Investigation #2 findings

### Task 1 — VesselManager._tick logic
- `_updatePositions(deltaYears)` iteruje wszystkie vessele
- Filtrowanie `mo?.status === 'active'` tylko dla `pursue`/`intercept` (skip `isOrderControlled`)
- Inaczej dispatch po `vessel.position.state`: docked / orbiting+m / orbiting+!m+dockedAt / **in_transit+m → INTERPOLACJA**
- **Branche NIE sprawdzają `movementOrder.status`** → cancelled moveToPoint nie był filtrowany

### Task 2 — MOS.cancelOrder side effects (V1 confirmed + ROZSZERZONY)
Przed P1.4.5: tylko `status, blockReason, _byVessel.delete, EventBus.emit`. Brak vessel.position/velocity/mission cleanup.

### Task 3 — `vessel:orderCancelled` subscribers (ROOT CAUSE)
**Jedyny subscriber:** `VesselManager._resumeMissionAfterOrder` — early-returns jeśli `vessel._suspendedMission === null`.

Plus `_suspendMissionIfAny` ma **dwa early-returns**:
- `if (!m) return false` (vessel bez wcześniejszej mission)
- `if (m.type === 'move_to_point') return false` (synth move_to_point nigdy nie snapshotowane)

**Konsekwencja:** moveToPoint issued na docked/idle vesselu → cancel → subscriber early-return → state pozostaje in_transit + active synth mission → vessel leci dalej.

### Task 4 — Arrival flow jako reference
`_onVesselArrived` (MOS:1067-1097) to **single source of truth** dla "moveToPoint completed cleanup":
```js
order.status = 'completed';
this._byVessel.delete(vessel.id);
vessel.mission = null;
vessel.status = 'idle';
// + VesselManager wcześniej:
//   vessel.position.state = 'orbiting'
//   vessel.position.dockedAt = null (dla synth m.targetId=null)
//   vessel.velocity zerowane
```

**Cancel powinien być symetryczny — same fields, ale `status='cancelled'` zamiast `'completed'`.**

### Task 5 — Test gap explicit
**Zero physics-level cancel testów od M1.** P1.4 testy sprawdzały tylko data-level (status='cancelled', EventLog push). M2b C4 testy sprawdzały tylko renderer filter (predictionCone). Po 9 miesiącach bug nadal niewykryty.

### `_stopVesselMotion` (NEW, full)

```js
/**
 * Cleanup pozycji/velocity/mission po cancel orderu.
 * Konwencja vessel.position.state ∈ {docked, orbiting, in_transit}; brak 'idle' —
 * "drift in space" reprezentujemy przez state='orbiting' + dockedAt=null
 * (spójne z _onVesselArrived dla moveToPoint, gdzie m.targetId=null).
 * Dla pursue/intercept oryginalna mission może być żywa (nie synth) — nie ruszamy
 * jej; jeśli był suspended snapshot, _resumeMissionAfterOrder podniesie state z
 * powrotem do 'in_transit' po naszym cleanup.
 */
_stopVesselMotion(vessel) {
  // Synth move_to_point mission — wywal całkowicie (zgodnie z _onVesselArrived).
  if (vessel.mission?.type === 'move_to_point') {
    vessel.mission = null;
  }
  vessel.position.state    = 'orbiting';
  vessel.position.dockedAt = null;
  vessel.status            = 'idle';
  const gameYear = window.KOSMOS?.timeSystem?.gameTime ?? 0;
  vessel.velocity.vx = 0;
  vessel.velocity.vy = 0;
  vessel.velocity.updatedYear = gameYear;
}
```

### L38 — Bug może żyć latami przez **test gap niewidocznego flow**

`MOS.cancelOrder` was tested data-level (status='cancelled') od M1 — devtools `KOSMOS.debug.cancelOrder` używany w dev/integration. ALE **nikt nie sprawdzał** `vessel.position.state` ani `velocity` post-cancel — bo "to oczywiste że jeśli order anulowany, to vessel się zatrzyma".

**Fundamentalny test gap:** smoke testy sprawdzały **input → data**, nie **input → physics**. Bug w **integracji** systems (MOS data ↔ VesselManager physics) był poza zakresem każdego pojedynczego testu.

**Plan mode mandate dla wszystkich M3+ commits:** pure helper testy są **necessary but not sufficient**. Każdy commit dotykający vessel/order/state lifecycle MUSI mieć **integration test** który:
1. Wywołuje API (np. cancelOrder)
2. Czeka na fan-out (events, subscribers)
3. Sprawdza **end-to-end state** (data + physics + UI/render side effects)

Plus diagnostyczna procedura dla user-facing features musi cytować **wszystkie related fields** w pre/post check (state, velocity, mission, status, orderStatus — nie tylko orderStatus).
