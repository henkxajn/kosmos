## Lekcje z P1.3.5 (kandydaci do M3 lessons learned)

### L28 — Gameplay intent per surface

Plan mode V verification "gdzie jest map / canvas" to za mało — trzeba zweryfikować **gameplay intent per surface** (która mapa jest "tv watch" vs "interactive command"). Architektura UI jest często warstwowana z wieloma surfaces o różnym intent — zakładanie "jest jedna mapa" = bug.

P1.2/P1.3 plan mode targetowały mapę 3D bo **ja** zinterpretowałem spec doc "right click map" jako mapę 3D. Filip's actual gameplay: mapa 3D = telewizor, tactical = command surface. Reframe wymagał osobnego commit (P1.3.5).

**Plan mode mandate dla P1.4+:** dla każdej kluczowej decyzji UI surface — **explicit confirm intent z Filipa** ("klikasz na X w mapie 3D czy w tactical map?").

### L29 — Spec doc apparent gap może być mismatch z runtime API

`KOSMOS.gameState.get('eventLog')` returned `undefined` → ja zakładałem **spec doc bug** ("M3 P3.1+ używa złej ścieżki"). Investigation Claude Code wykrył że spec doc **już używał poprawnego API** (`KOSMOS.eventLogSystem`) — moja runtime probe była błędna ścieżka.

**Plan mode mandate:** PRZED założeniem "spec doc bug" — `grep` literalnie w spec doc czy użyje złej ścieżki. Mismatch między mental model a runtime API ≠ spec bug.

### L30 — System X może już istnieć (grep PRZED założeniem)

`EventLogSystem` istniał od M2b (klasa kompletna z public API: push, getEntries, getVisible, toggleChannel). Ja go nie skojarzyłem przy pisaniu M3 spec. **Plan mode V_extra dla pure helperów cytujących system X**: `grep -rn "class.*X\|new X" src/` PRZED założeniem ścieżki access.

### L31 — Pure helpers konsumujące outputs z multiple sources muszą obsługiwać multiple shapes

OrderDispatcher był pisany dla raycaster z mapy 3D (XZ plane → `{x, y, z}`). Tactical raycaster w P1.3.5 zwraca `{x, y}` (2D). **Single-shape assumption** → `no_target_point` runtime reject (smoke pass był false-positive bo testował tylko 3D shape).

**Pattern:** pure helper consuming `worldPoint` musi mieć **explicit type discriminator** lub **multi-shape compatibility** z null guards. Comment z reference to multiple producers.

### L32 — D2 "disable handlers" wymaga target-zależnego check, nie stan-zależnego

P1.3.5 base D2 = `if (overlayManager.isAnyOpen()) return` — disabled mapa 3D **tylko** gdy FleetOverlay open. Z FleetOverlay zamkniętym → mapa 3D dispatchowała orders jak P1.3 base.

**Lekcja:** "disable X gdy Y" wymaga że Y = condition która **zawsze** zachodzi w docelowym scope. Jeśli Y jest stan-zależne (overlay open/closed) — fail dla edge cases. Lepszy pattern: target-zależny check (`e.target.id === ...`) który nie zależy od stanu.

### L33 — Tag PRZED real-flow = pre-emptive promise

Claude Code dał tag `m3-commit-p1-3.5-tactical` na commit base **przed** Filip pass real-flow. Workflow inconsistency — taki tag = symbol pass smoke testów, nie real-flow. Filip wykonał real-flow → T-E fail → post-fix wymagany ALE tag już aktywny.

W P1.3.5 zostało to OK (tag zachowany na 0c720d9, post-fix to delta po nim). Ale **lekcja**: tag to symbol pass real-flow Filipa. Tag PRZED real-flow = pre-emptive promise. Powstrzymywać Claude Code od pre-emptive tagowania.

## Konkretny diff (post-fix #1 critical changes)

### `OrderDispatcher.js` (linia ~21):

```js
// Pure helper — konwertuje worldPoint do targetPoint dla MOS.
// Obsługuje OBIE konwencje:
//   - L8 (3D mapa, XZ plane Y=0): wp = {x, y, z}, używamy wp.z jako y
//   - P1.3.5 tactical (2D natywne): wp = {x, y}, używamy wp.y bezpośrednio
function POINT_FROM_WORLD(wp) {
  if (!wp || typeof wp.x !== 'number') return null;
  const y = typeof wp.z === 'number' ? wp.z : wp.y;
  if (typeof y !== 'number' || !Number.isFinite(y)) return null;
  return { x: wp.x, y };
}
```
